SEC 01 ── ABSTRACT
════════════════════════════════════════════════════════════════════════

Make slot duration a runtime configuration. Then reduce it.

This EIP turns SLOT_DURATION_MS from a compile-time constant into a fork-activated runtime parameter on the consensus layer, then uses that infrastructure to reduce slot duration from 12 seconds to 8 seconds. Block gas limits and blob parameters scale proportionally to maintain constant throughput per unit time. 

SEC 02 ── THE PROBLEM
════════════════════════════════════════════════════════════════════════

Twelve seconds is slow.

Slot time is the heartbeat of Ethereum's user experience. Every second shaved off means faster transaction landings, faster exchange deposits, and faster real-world payments. But the costs of long slots extend well beyond UX — they inflate arbitrage losses, widen the MEV extraction window, grant builders an exploitable free option on blocks, and force the ecosystem to build complex preconfirmation protocols to paper over the latency.

  12,000 ms                              8,000 ms

SEC 04 ── APPROACH
════════════════════════════════════════════════════════════════════════

Nobody knows the safe minimum slot duration with today's client implementations. Rather than stalling on the choice of a number, the work is separated into three phases. 

  PHASE 1                   PHASE 2                   PHASE 3
  INFRASTRUCTURE      ───►  CHARACTERIZE        ───►  REDUCE

  ┌───────────────────────┐ ┌───────────────────────┐ ┌───────────────────────┐
  │                       │ │                       │ │                       │
  │ Remove the 12-second  │ │ Systematically        │ │ Cut slot duration     │
  │ assumption across all │ │ identify CL           │ │ based on empirical    │
  │ clients. Make         │ │ bottlenecks through   │ │ results. Address      │
  │ SLOT_DURATION_MS a    │ │ devnets & benchmarks, │ │ client constraints,   │
  │ fork-activated        │ │ analogous to the EL's │ │ reduce further as     │
  │ runtime config.       │ │ bloat-nets and        │ │ headroom emerges.     │
  │                       │ │ perf-nets.            │ │ Iterate.              │
  │                       │ │                       │ │                       │

SEC 05 ── SPECIFICATION
════════════════════════════════════════════════════════════════════════

At <FORK_EPOCH>, the following constants take effect. The general principle: do not adjust a constant unless there is a concrete security or economic failure from leaving it unchanged. 

┌─────────────────────────────────┬────────────┬─────────────┬──────────────────────────────┐
│ PARAMETER                       │ CURRENT    │ NEW         │ RATIONALE                    │
├─────────────────────────────────┼────────────┼─────────────┼──────────────────────────────┤
│ SLOT_DURATION_MS                │ 12,000     │ --> 8,000   │ Target slot duration         │
│ BASE_REWARD_FACTOR              │ 64         │ --> 42      │ Preserve annual issuance     │
│ INACTIVITY_PENALTY_QUOTIENT     │ 16,777,216 │ --> 37,748,736 │ Quadratic penalty scaling │
│ MIN_EPOCHS_FOR_BLOB_SIDECARS    │ 4,096      │ --> 6,144   │ Preserve wall-clock DA window│
│ CHURN_LIMIT_QUOTIENT            │ 65,536     │ --> 98,304  │ Preserve weak subjectivity   │
└─────────────────────────────────┴────────────┴─────────────┴──────────────────────────────┘

Gas limit scales by new_slot_duration / old_slot_duration at the fork block. Normal gas limit voting resumes from the next block. Blob parameters scale proportionally via a new BLOB_SCHEDULE entry.
