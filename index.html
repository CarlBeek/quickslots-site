SEC 01 ── ABSTRACT
════════════════════════════════════════════════════════════════════════
╔══════════════════════════════════════════════════════════════════════╗
║     Make slot duration a runtime configuration. Then reduce it.      ║
╚══════════════════════════════════════════════════════════════════════╝

This EIP turns SLOT_DURATION_MS from a compile-time constant into a fork-activated runtime parameter on the consensus layer, then uses that infrastructure to reduce slot duration from 12 seconds to 8 seconds. Block gas limits and blob parameters scale proportionally to maintain constant throughput per unit time. 

SEC 02 ── THE PROBLEM
════════════════════════════════════════════════════════════════════════
╔══════════════════════════════════════════════════════════════════════╗
║                       Twelve seconds is slow.█                       ║
╚══════════════════════════════════════════════════════════════════════╝

Slot time is the heartbeat of Ethereum's user experience. Every second shaved off means faster transaction landings, faster exchange deposits, and faster real-world payments. But the costs of long slots extend well beyond UX — they inflate arbitrage losses, widen the MEV extraction window, grant builders an exploitable free option on blocks, and force the ecosystem to build complex preconfirmation protocols to paper over the latency.

Arbitrage losses scale with the square root of inter-block time [4]. Moving from 12 to 8 seconds yields roughly 18% tighter on-chain pricing. Shorter slots also shrink the builder's free option on blocks [5], reducing the incentive to submit empty blocks under ePBS.

SEC 04 ── APPROACH

Six seconds would be ideal, but given this work is designed as a non-headliner alongside FOCIL (EIP-7805), eight seconds is the realistic first target. Even ten seconds would deliver a meaningful improvement. The variable slot timing infrastructure is largely orthogonal to FOCIL, so both advance in parallel.

SEC 05 ── SPECIFICATION
════════════════════════════════════════════════════════════════════════

At FORK_EPOCH, the following constants take effect. The general principle: do not adjust a constant unless there is a concrete security or economic failure from leaving it unchanged. 

┌──────────────────────────────────┬────────────────┬──────────────────┐
│ PARAMETER                        │ CURRENT        │ NEW              │
├──────────────────────────────────┼────────────────┼──────────────────┤
│ SLOT_DURATION_MS                 │        12,000  │          8,000   │
│ BASE_REWARD_FACTOR               │            64  │             42   │
│ INACTIVITY_PENALTY_QUOTIENT      │    16,777,216  │     37,748,736   │
│ MIN_EPOCHS_FOR_BLOB_SIDECARS     │         4,096  │          6,144   │
│ CHURN_LIMIT_QUOTIENT             │        65,536  │         98,304   │
└──────────────────────────────────┴────────────────┴──────────────────┘
┌──────────────────┬────────────────┬────────────────┬─────────────────┐
│                  │ CURRENT (12s)  │ PROPOSED (8s)  │ CHANGE          │
├──────────────────┼────────────────┼────────────────┼─────────────────┤
│ GAS / BLOCK      │    36,000,000  │    24,000,000  │ -33%            │
│ BLOBS / BLOCK    │             6  │             4  │ -33%            │
│ GAS / SECOND     │     1,500,000  │     1,500,000  │ = UNCHANGED     │
│ BLOBS / SECOND   │           0.5  │           0.5  │ = UNCHANGED     │
│ FINALITY         │       ~13 min  │     ~8.5 min   │ -33%            │
└──────────────────┴────────────────┴────────────────┴─────────────────┘

Gas limit scales by new_slot_duration / old_slot_duration, enforced at the fork block rather than left to validator voting. Without enforcement, the +/-1/1024-per-block voting mechanism would take ~45 minutes to converge, during which gas-per-second throughput could overshoot the target by up to 50%. Normal voting resumes from the next block. Blob parameters scale proportionally via a new BLOB_SCHEDULE entry.

Requires a hard fork. CL clients bear most of the change: replacing hardcoded 12-second slot assumptions with fork-aware timing derivations. EL impact is limited to a one-time gas limit and blob parameter adjustment at the fork boundary. 

SEC 06 ── RATIONALE
════════════════════════════════════════════════════════════════════════
┌─ GUIDING PRINCIPLE ─────────────────────────────────────────────────┐
│                                                                      │
│  Do not adjust a constant unless there is a concrete security or     │
│  economic failure from leaving it unchanged.                         │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘

Most slot- and epoch-denominated constants have generous margins that absorb the change without adjustment. For example, EPOCHS_PER_SLASHINGS_VECTOR shrinks from ~36 to ~24 days but remains far longer than any plausible attack window. Four categories do require scaling:

Issuance: BASE_REWARD_FACTOR scales linearly with epoch duration to preserve annualized yield. The integer truncation (42 vs. the ideal 42.667) under-issues by ~1.6%, less than typical participation-rate fluctuations.

Inactivity leak: the cumulative penalty is quadratic in epoch count. The quotient scales by the square of the epoch-duration ratio to preserve the wall-clock leak rate.

Data availability: rollup challenge periods (~7 days) create a hard external dependency. Sidecar retention windows scale inversely to preserve the same wall-clock duration.

Churn limits: per-epoch churn limits scale to preserve the wall-clock weak subjectivity period. Quotients (which are divisors) scale by old/new.
